Sabse pehle ek baat yaad rakho. **`select_related()` koi naya data nahi lata.** Ye sirf **data lane ka tareeka (optimization)** change karta hai. Matlab: * Output same rahega. * Data same rahega. * Sirf database se data lane ka method change hoga. --- # Real Life Example Maan lo ek college hai. Student table | id | Name | Department_id | | -- | ----- | ------------- | | 1 | Rahul | 1 | | 2 | Riya | 1 | | 3 | Mohit | 2 | Department table | id | Department | | -- | ---------------- | | 1 | Computer Science | | 2 | Mechanical | Ab Rahul ke paas department ka naam directly store nahi hai. Uske paas sirf ``` department_id = 1 ``` hai. Department ka actual naam doosri table me pada hua hai. Isi ko ForeignKey bolte hain. --- # Ab Django kya karta hai? Aapne likha ```python students = Student.objects.all() ``` Abhi tak Django ne sirf Student table padhi. Uske paas ye data aaya ``` Rahul Riya Mohit ``` Bas. Department ka naam abhi nahi aaya. --- Ab aapne loop chalaya ```python for student in students: print(student.name) ``` Output ``` Rahul Riya Mohit ``` Sab sahi. --- Ab aapne likha ```python for student in students: print(student.name, student.department.name) ``` Ab dikkat shuru. Rahul ke liye Django sochta hai > "Mujhe Rahul ka department bhi chahiye." To database me fir se jata hai. ``` SELECT Department WHERE id=1 ``` Rahul complete. --- Ab Riya Uska bhi department chahiye. Fir database gaya. ``` SELECT Department WHERE id=1 ``` Dobara. --- Ab Mohit ``` SELECT Department WHERE id=2 ``` Fir database. --- # Matlab Student table ``` Ek baar ``` Department table ``` Teen baar ``` Total ``` 4 baar database gaya. ``` Yahi problem hai. Isko **N+1 Query Problem** bolte hain. --- # Ye problem hoti kyun hai? Kyuki Django lazy hai. Wo bolta hai > "Jab tak tum department nahi maangoge, main nahi launga." Isliye jab aap ``` student.department ``` likhte ho tabhi Django database ke paas bhagta hai. --- # Ab Entry Hoti Hai select_related() Ki Aapne likha ```python students = Student.objects.select_related("department") ``` Ab Django pehle hi samajh gaya. Usne socha > "Mujhe pata hai baad me department use hoga." To usne pehle hi dono tables ko jod diya. Student table * Department table Ek hi query me. --- # Andar kya hota hai? Database me kuch aisa hota hai ``` Student JOIN Department ``` Ye JOIN SQL ka feature hai. Isliye database sirf ``` Ek baar ``` chalti hai. --- # Fir loop ```python for student in students: print(student.name, student.department.name) ``` Ab Django database ke paas nahi jata. Kyuki department pehle hi memory me pada hua hai. --- # Dhyan Do Normal ``` Student ↓ Rahul ↓ Database ↓ Department ``` Fir ``` Riya ↓ Database ↓ Department ``` Fir ``` Mohit ↓ Database ↓ Department ``` Har student ke liye database. --- select_related() ``` Student + Department ↓ Ek hi baar ↓ Memory ``` Ab jitni baar chahe ``` student.department.name ``` likho. Database nahi jayega. --- # Kab Use Karen? Sirf jab relation ho ``` ForeignKey ``` Ya ``` OneToOneField ``` Jaise Student ↓ Department Teacher ↓ Department Student ↓ Profile Subject ↓ Teacher Ye sab `select_related()` ke cases hain. --- # Kab Nahi Karna? Agar relation ``` ManyToMany ``` ho. Jaise ``` Subject ↓ Students ``` Ek subject me bahut students. Ek student me bahut subjects. Yahan JOIN bahut bada aur complex ho sakta hai. Isliye Django `select_related()` allow hi nahi karta. Yahan `prefetch_related()` use hota hai. --- # Sabse Easy Rule Agar relation **"Ek object se sirf ek object"** tak ja raha hai, to `select_related()`. Examples: * Student → Department ✅ * Student → Profile ✅ * Subject → Teacher ✅ * Teacher → Department ✅ Agar relation **"Ek object se bahut saare objects"** tak ja raha hai, to `prefetch_related()`. Examples: * Department → Students * Teacher → Subjects * Subject → Students --- # Ek Sentence Mein **`select_related()` Django ko pehle hi bol deta hai: "Mujhe related ForeignKey ya OneToOne ka data bhi chahiye, isliye sab ek hi query me lekar aao."** Isse output nahi badalta, sirf database ke chakkar kam ho jaate hain, aur application fast ho jaati hai. Bilkul sir. Ab **`prefetch_related()`** ko bhi wahi style me samajhte hain—**Hinglish + Deep + Real Life + Zero Ratta**. --- # `prefetch_related()` Kya Hai? Sabse pehle ek baat yaad rakho. Jaise `select_related()` output change nahi karta tha, waise hi **`prefetch_related()` bhi output change nahi karta.** Ye bhi **sirf data lane ka tareeka optimize karta hai.** --- # Sabse Pehle Real Life Maan lo hamare paas ye data hai. ## Student | id | Name | | -- | ----- | | 1 | Rahul | | 2 | Riya | | 3 | Mohit | --- ## Subject | id | Name | | -- | ------ | | 1 | Python | | 2 | Java | | 3 | Django | --- ## Subject_Students (ManyToMany Table) | Subject | Student | | ------- | ------- | | Python | Rahul | | Python | Riya | | Python | Mohit | | Java | Rahul | | Java | Mohit | | Django | Riya | Ye table Django khud banata hai jab aap `ManyToManyField` use karte ho. --- # Ab Hum Query Likhenge ```python subjects = Subject.objects.all() ``` Abhi Django ne sirf Subject table padhi. Memory me ```text Python Java Django ``` Bas. Students ka data abhi nahi aaya. --- # Ab Loop Chalaya ```python for subject in subjects: print(subject.name) ``` Output ```text Python Java Django ``` Sab sahi. --- # Ab Humne Likha ```python for subject in subjects: print(subject.name) for student in subject.students.all(): print(student.name) ``` Ab asli game shuru hoti hai. --- # Django Sochta Hai ### Python > "Python subject ke students chahiye." Database gaya. ```text SELECT students WHERE subject = Python ``` Mil gaya. ```text Rahul Riya Mohit ``` --- ### Ab Java Fir database gaya. ```text SELECT students WHERE subject = Java ``` --- ### Ab Django Fir database gaya. ```text SELECT students WHERE subject = Django ``` --- # Total Subject table ```text 1 Query ``` Students ```text Python ↓ 1 Query ``` Java ↓ ```text 1 Query ``` Django ↓ ```text 1 Query ``` Total ```text 4 Queries ``` Agar 100 Subjects to 101 Queries. Yahi N+1 Problem hai. --- # select_related() Kyu Nahi? Bahut important question. Kya hum likh sakte hain? ```python Subject.objects.select_related("students") ``` Answer ❌ Nahi. Error aa jayega. Kyuki Ek Subject ke ```text Rahul Riya Mohit Anjali Karan ``` Kitne bhi students ho sakte hain. Database JOIN bahut bada ho jayega. Isliye Django mana kar deta hai. --- # Ab Entry Hoti Hai prefetch_related() Ki Aapne likha ```python subjects = Subject.objects.prefetch_related("students") ``` Ab Django smart ho gaya. Usne socha > "Mujhe pata hai baad me students use honge." To usne pehle hi plan bana liya. --- # Django Kya Karta Hai? Pehli Query ```text Saare Subjects le aao. ``` Database ↓ ```text Python Java Django ``` --- Fir Doosri Query ```text In tino subjects ke jitne bhi students hain, ek hi baar me le aao. ``` Database ↓ ```text Rahul Riya Mohit ``` Sab ek saath. --- Fir Django khud memory me mapping bana deta hai. Jaise ```text Python ↓ Rahul Riya Mohit ``` Java ↓ ```text Rahul Mohit ``` Django ↓ ```text Riya ``` Ab jab aap loop chalate ho ```python for subject in subjects: print(subject.name) for student in subject.students.all(): print(student.name) ``` Django database ke paas nahi jata. Wo memory se hi students nikal deta hai. --- # Dhyan Se Difference Dekho ## Without prefetch_related() ```text Subject ↓ Python ↓ Database ``` Fir ```text Java ↓ Database ``` Fir ```text Django ↓ Database ``` Har subject ke liye database. --- ## With prefetch_related() ```text Database ↓ Saare Subjects ``` Fir ```text Database ↓ Saare Students ``` Bas. Ab jitni baar chahe loop chalao. Database nahi jayega. --- # Isliye Naam Hai "Prefetch" **Pre** = Pehle **Fetch** = Data lana Matlab > "Jo related data baad me chahiye hoga, use pehle hi lekar aa jao." --- # Kab Use Karen? Jab relation ho * ManyToManyField ✅ * Reverse ForeignKey ✅ Jaise ```python Student.objects.prefetch_related("subjects") ``` Ek student ke bahut subjects. --- ```python Subject.objects.prefetch_related("students") ``` Ek subject ke bahut students. --- ```python Department.objects.prefetch_related("students") ``` Ek department me bahut students. --- ```python Teacher.objects.prefetch_related("subjects") ``` Ek teacher bahut subjects padha sakta hai. --- # Sabse Easy Trick Yaad Rakhna ### `select_related()` Socho: 👤 **Ek Student → Ek Department** Ek se Ek. To **JOIN** ho sakta hai. Isliye `select_related()`. --- ### `prefetch_related()` Socho: 📚 **Ek Subject → 500 Students** Ek se Bahut. JOIN bahut bada aur duplicate data la sakta hai. Isliye Django bolta hai: > "Main do alag queries chalaunga, phir Python me unhe jod dunga." Yehi `prefetch_related()` hai. --- # Golden Rule (Interview Trick) Kabhi confuse ho to sirf ye sochna: * **Ek object ke saath sirf ek related object?** → `select_related()` * **Ek object ke saath bahut saare related objects?** → `prefetch_related()` --- # Ek Line Mein **`prefetch_related()` Django ko pehle hi bol deta hai: "ManyToMany ya reverse relationship ka related data bhi pehle hi alag query me fetch kar lo, phir memory me connect kar dena."** Isi wajah se output bilkul same rehta hai, lekin database ke unnecessary rounds bach jaate hain aur application ka performance kaafi improve ho jata hai.
